SageJiraBot — About & How to Use

Turn a meeting transcript into a batch of reviewed Jira tickets in about 60 seconds.

What it does

SageJiraBot runs a multi-agent pipeline that reads a meeting transcript, extracts the action items, filters out chatter, writes clean Jira-style descriptions, and presents the result as a reviewable draft table before anything is sent to Jira. You edit, uncheck, or adjust any row; then one click creates the whole batch.

Getting Started

  1. Sign in to Jira (per-user OAuth). In the left sidebar under “Step 1 — Connect to Jira” there’s a Jira Sign-In (per-user) card with a blue button. Click it. You’ll be redirected to Atlassian, asked to approve the scopes, then sent back signed in. Your tickets will be created as you, not a shared service account.
  2. Pick a Jira project. Once signed in the project dropdown populates automatically. By default it shows projects you’ve recently viewed (typically 10–20). Tick “Show all projects I can write to” if the one you want isn’t listed.
  3. Provide a transcript in Step 2. Drag in a .txt, .vtt, or .srt file, or paste text directly into the textarea. Teams recap transcripts and Webex .txt exports work as-is.
  4. Click “Analyze Transcript.” The three-agent pipeline runs in the background; in about 30–60 seconds the draft table appears in Step 3.
  5. Review, edit, then submit. Uncheck rows to drop them; edit any field inline; use the bulk-apply bar to set epic/sprint/assignee on many rows at once; click “Create Selected Tickets.”

How the Pipeline Works (under the hood)

You don’t need to know this to use the app, but it explains why some tickets are dropped and what the AI is looking for.

  • Agent 1 (extractor) reads the transcript and emits a structured list of candidate tickets: summary, description, effort hint, priority hint, any names mentioned near each task.
  • Agent 2 (router) filters to actionable items. Things that are just discussion, background, or rhetorical (“we should think about X someday”) get filtered out. What Agent 2 rejects is shown in the collapsible “N items filtered out by Agent 2” panel above the draft table — expand it to sanity-check the agent isn’t dropping things you care about.
  • Agent 3 (writer) rewrites each approved item into a Jira-ready summary and description with proper acceptance-criteria shape.

Guiding the Conversation So Transcripts Extract Cleanly

Upload whatever transcript you get — the bot never asks you to edit it before upload. These are habits to use during the meeting that make the raw captured transcript easier for the AI to turn into clean tickets.

State action items out loud

  • Say “Action item: X will do Y by Z” or equivalent when an item is decided. The word “action item” is a strong signal for Agent 1.
  • Use imperative verbs (“Refactor the login flow”, “Add a CSV export”) rather than vague framings (“we were thinking about maybe looking into refactoring”).

Say the assignee's name near the task

  • “Joe will handle the migration script.” → the draft is pre-assigned to Joe (if there’s only one Joe on the project).
  • “The migration script needs work. Someone should do it.” → no name linked; assignee stays empty.

Name matching tries exact display name first, then email local-part (“jsmith” → jsmith@duke.edu), then a unique first-name or last-name match. When it’s ambiguous (two people named Joe on the project) the draft is left unassigned with a grey “Suggested: Joe” hint so you pick the right one from the dropdown.

Call out deadlines and priority in plain words

  • “by next Friday”, “in the next sprint”, “before the Aug release” give Agent 1 something to base the start/due date on.
  • “This is a blocker”, “production is down”, “critical for the demo” → priority gets bumped up.
  • Without those signals the default priority is Medium and dates are estimated from effort + dependencies.

Finish one topic before jumping to the next

Conversations that ping-pong (“oh and while we’re talking about that, also the other thing, and also Jim mentioned…”) produce overlapping tickets because the AI can’t tell where one item ends and the next begins. When you notice that pattern, pause and name the current topic explicitly (“okay, next topic — deployment…”) before moving on. It costs nothing in the meeting and saves manual ticket-splitting later.

Step 3 — The Draft-Tickets Screen

Per-row editing

Every column in the draft table is editable: Summary, Epic, Effort, Priority, Start, Due, Blocks, Sprint, Fix Version, Labels, Assignee, Description. Click a cell and type. Changes are held in the browser until you click Create Selected Tickets.

Uncheck the leftmost checkbox to drop a row from the batch. The Select all checkbox in the header toggles every row.

The anchor-date bar (scheduling)

Above the table: Anchor Date (project start) + Recalculate Dates button. This sets the first ticket’s start date; everything else is computed forward from effort estimates and dependency chains. Change the anchor to slide the whole schedule forward or back without editing each row individually.

The bulk-apply bar

Above the table: Apply to all checked rows. Use this after analysis to set a field across many rows in one click. Fields supported:

FieldWhat it does
Epic Pick an existing epic from the project, or type a new name into the “New epic name...” box that appears — a new Epic ticket is created first and all checked rows land under it.
Sprint Assign every checked row to an active or upcoming sprint (Scrum boards only; Kanban boards show nothing here).
Fix Version Mark every checked row for a specific release version.
Assignee Assign every checked row to one person. Dropdown is populated with actual project role members (not the full Atlassian org).
Priority Force Critical / High / Medium / Low on every checked row.
Start Date / Due Date Set the same start or due date across all checked rows (useful when a group of tasks is bundled in one sprint).

Workflow: uncheck rows you don’t want affected → pick values in the bulk bar → click Apply. You can apply the bar multiple times in sequence (e.g. set Epic for all, then check a subset and set Assignee).

Dependencies and “Blocks”

Tickets that depend on other tickets from the same batch show up in the Blocks column. A ticket that blocks another affects the auto-computed dates (scheduling respects the chain). When created, these become real Jira “blocks” links between the issues.

Default labels on every created ticket

  • sageJiraBot — identifies tickets created by this tool, useful for later filtering in JQL.
  • <batch-timestamp> — a timestamp like 20260422-2158 groups all tickets from one submission together. Run a JQL like labels = "20260422-2158" to find (or bulk-delete) just that batch.

Add your own labels via the Labels column on each row. These merge with the two defaults above; the defaults cannot be removed.

Who Tickets Are Created As

If you signed in via the Jira Sign-In (per-user) button: every ticket’s Reporter is you. Audit history is accurate, team members can see who filed what, and notification rules in Jira trigger based on your identity.

If you didn’t sign in (rare; the UI now requires it): tickets are created by the service-account API token, which attributes everything to a single shared identity. Not ideal for teamwork, but fine for automated pipelines.

Planned Features (Not Yet Available)

Features that are designed but not yet wired up. Listed here for transparency; do not rely on them in a live demo.

  • Natural-language directives during the meeting. The idea: phrases like “assign all of these to Joe”, “make this a blocker”, or “skip that one” would be picked up by Agent 2 and applied to the drafts before you see them. Today the router ignores imperatives like this — you set assignees and priorities via the bulk-apply bar in Step 3. (Roadmap: extend core/agents/agent2_router.py with a second pass that looks for directive sentences near each candidate ticket.)
  • Incoming email ingestion. Watching the DCRI shared error mailbox and auto-running the pipeline on new messages. Blocked on a Microsoft Graph Azure AD app registration (DCRI IT ticket DIS-3565, submitted 2026-04-22).
  • Separate “AI estimate” field in Jira. A Jira custom field that stores the AI’s original hour estimate alongside the human’s edited one, for post-hoc accuracy analysis. Blocked on Jira admin creating the field (DCRI IT ticket DIS-3564, submitted 2026-04-22).

Troubleshooting Quick Reference

SymptomLikely cause / fix
“Sign in to Jira” opens a new tab showing “couldn’t identify the app” Client ID misconfiguration. Ask the server operator to verify ATLASSIAN_OAUTH_CLIENT_ID matches the value on developer.atlassian.com.
Signed in, but the project dropdown shows all ~500 projects Duke’s Jira grants Create permission broadly. The default filter (“recently viewed”) needs some Jira activity from your account to populate; if it has none, the toggle shows all writable. Use Jira normally for a few days and the list narrows.
Analyze fails / times out Usually AI Foundry is unreachable (VPN dropped). Test with curl -I https://ai-foundry-dcri-sage.cognitiveservices.azure.com/ from the server.
Ticket creation succeeds but epic link is missing The Epic chosen in the bulk bar might have been archived or renamed. Pick from the dropdown (don’t type a free-form key).
I want to find every ticket this bot ever created, or roll one batch back Two JQLs to know:
labels = "sageJiraBot" — every ticket the bot has ever created (across all batches and all users).
labels = "<batch-timestamp>" — just the batch from one submission (the timestamp like 20260422-2158 is shown in the header of Step 3 after submission). Combine with reporter = currentUser() to narrow to your own.
